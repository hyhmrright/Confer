# Wave 1 用户访谈与日记研究脚本（C-E1 + B-E1）

本脚本同时执行两项 Wave 1 desirability 实验：**C-E1**（开发者 IDE 内跨人 tacit-knowledge 时刻日记 + 访谈，验证 C-v1）与 **B-E1**（agent 框架开发者 counterparty-reality 访谈 + 流量审计，验证 B-v1）。两者共享招募基础设施、反偏差守则与伦理流程；日记协议只属于 C-E1，流量审计只属于 B-E1。

本研究**只产出 stated/recalled 证据（受访者回忆 + prod 流量自报），不产出 revealed-preference 行为证据，更不产出「两个独立人完成 agent-to-agent 往返」的证据**。后者分别是 C-E2（Wizard-of-Oz）与 C-E3（pre-registered 双非创始人门）的职责。把本研究的产出当成「bet 通过」是范畴错误——本脚本反复标注此边界。

---

## 0. NON-FOUNDER GATE（贯穿全脚本的硬规则）

> **全局规则：** 创始人用自己两个账号互发消息只证明「管子通了」，**不**证明 bet。整个 portfolio 的成功只有在 **≥2 个独立的、非创始人的真人** 完成 unscripted、consequential 的 agent-to-agent 往返时才计数（这正是 C-E3 pre-register 的硬门）。
>
> **本研究的角色与边界：** C-E1/B-E1 都是 desirability 访谈/日记研究，**本身不构成 bet 通过的证据**，也不应被任何下游决策当成已通过 NON-FOUNDER GATE。它们的唯一交付是「干净的、非创始人为主的需求信号」，喂给 C-E2/C-E3。因此本研究的每条记录都打 `founder_proximity` 标（见 §2.4）：**创始人本人及其直接圈层（同事/合伙人/好友）的记录单独隔离、永不进入任何阈值分母**，否则会向下游喂入污染样本，制造 C-E3 想防的那种 false positive。
>
> 每份交付物（§8.5）必须显式声明：「本研究未、且不能单独验证全局 bet；bet 的通过判定属 C-E3。」

## 1. 研究目标与对应假设

本研究是 desirability-before-feasibility 阶段的最前置一跳：先证明「跨人需求真实存在」，再谈路由行为（C-E2）、付费（C-E4/B-E4）、分发（C-E5）。

### 1.1 C-E1 目标 → 假设映射

| 研究问题 | 映射假设 ID | 该问题要证伪/证实的 |
|---|---|---|
| 开发者在 IDE 内是否真有**反复出现**的、需要**某个特定他人**的 tacit knowledge 的时刻？ | **C-v1**（value, rank 1） | 全局 bet 在 IDE 场景的本地实例化——若不成立，C 的可行性/盈利/分发假设全部 moot |
| 这些需求是否**不是** code/library/repo（模型可解）类，而是真·跨人 tacit（决策理由、谁拥有 X、access/credentials owner、领域专家口头判断）？ | **C-v1** | devil's-advocate 的核心：IDE 问题大多是模型强项，真人需求通常 Slack 更快 |
| 对这些真·跨人需求，**把请求路由给那个人的 agent** 是否比直接 Slack DM 那个人更快/更好？ | **C-v1**（反事实部分） | 即使需求存在，是否能赢过「那个人就在 Slack 上」这一现成替代 |
| 创始人本人圈内的回声是否被审计排除？ | **NON-FOUNDER GATE / shared assumption** | 防止把易得的、最不具信息量的版本当成信号污染下游 |

### 1.2 B-E1 目标 → 假设映射

| 研究问题 | 映射假设 ID | 该问题要证伪/证实的 |
|---|---|---|
| agent 框架开发者今天是否真有**生产中**、与**自己信任边界之外**的 agent/endpoint 交互的实例？ | **B-v1**（value, rank 1） | 若几乎全是 intra-trust（自家 agent 调自家工具 / 已知 SaaS API key），SDK 是「等市场的方案」 |
| 在该交互里，**可验证的跨 agent 身份**是否被点名为**首要** blocker（而非 capability/liability/output 质量）？ | **B-v1** + **shared identity assumption** | identity 是否真是 binding constraint，还是 necessary-but-not-sufficient |
| 把所有 blocker 列全并归类，**非身份类 blocker 是否占被点名 blocker < 50%**？ | **B-v1**（successCriteria 第二条） | 一次审计同时重构 B/D/F/H 四个 idea 的身份前提 |

## 2. 招募与筛选（screener）

### 2.1 两条受访者管线

| 管线 | 目标人群 | 目标样本量（来自 successCriteria 分母） | 渠道 |
|---|---|---|---|
| **C-E1** | 每日使用 Claude Code 或 Cursor 的开发者 | **12–15 人**（阈值用 6/15 ≈ 40%） | 创始人网络 + 2–3 个 dev Slack/Discord 社区 |
| **B-E1** | 在 prod 或 near-prod 运行 LangChain/CrewAI/MCP-host/AutoGen 的 agent builder | **20 人**（阈值用 8/20 = 40%） | dev Discord、MCP/A2A GitHub issue 线程、r/LocalLLaMA、创始人 Confer 联系人 |

> **范畴提醒：** C-E1 招的是「用 host IDE 的普通开发者」，B-E1 招的是「自己写 agent 框架代码的 builder」——两类人重叠少，**不要混用同一招募帖**。

### 2.2 C-E1 合格条件（screener，全部满足才合格）

1. 过去 **7 天**内，**每个工作日**都在 Claude Code 或 Cursor 中实际写过代码（自报 + 让其口述昨天具体干了什么以反诈）。
2. 工作中需要**与其他人协作**（有同事/队友/外部协作方），即存在「可能需要别人脑子里的东西」的结构条件。
3. 愿意做 **3 天**轻量日记 + **30 分钟**访谈。
4. **不是**纯个人 side-project 单干者（单干者不存在跨人需求，会系统性拉低真阳率但属合理排除——记录排除原因）。

### 2.3 B-E1 合格条件（screener，全部满足才合格）

1. 当前有 agent 跑在 prod 或 near-prod（不是纯本地 demo）——让其说出框架名 + 大致流量场景。
2. 其 agent **调用过它不掌控的东西**（别处的 endpoint / 第三方 agent / 外部 API）——这是 counterparty 是否存在的前置过滤，但 **screener 阶段不暗示「身份」是问题**，只问「调没调过你不掌控的东西」。
3. 愿意做 **30 分钟** problem interview（**纯发现，无 pitch**）。

### 2.4 排除创始人圈内偏差（强制，NON-FOUNDER GATE 的执行点）

每条受访记录打 `founder_proximity` 标，三档：

| 标 | 定义 | 处理 |
|---|---|---|
| `direct` | 创始人本人、现/前同事、共同创业者、私人好友 | **隔离**：不进入任何阈值分母；仅作背景/试运行/脚本校准 |
| `network_1hop` | 由创始人直接介绍、或创始人社区中与创始人有过 1:1 互动的人 | 进入分母，但**单独子样本统计**，并在报告里给出「去掉 1hop 后阈值是否仍成立」的敏感性检查 |
| `cold` | 公开社区帖、marketplace、issue 线程冷招募，与创始人无个人关系 | 主力分母，权重最高 |

招募硬约束（防 vanity confirmation）：
- C-E1 与 B-E1 各自的 **`cold` 占比目标 ≥ 1/3**（即 C-E1 至少 5 人 cold、B-E1 至少 7 人 cold）。若达不到，先扩冷招募，不要用 `direct` 凑数。
- 招募文案**不得**包含 idea 名、产品名、「agent 互相说话」「让你的 agent 替你联系别人」等暗示性价值主张（见 §6 文案守则）。
- 同一团队招募上限：C-E1 同公司 ≤ 2 人、B-E1 同公司 ≤ 2 人（防单一团队的局部习惯被当成普遍模式）。

### 2.5 招募外联模板（中性、无产品暗示）

**C-E1 招募原话：**
> 我在做一项关于「开发者在 IDE 里实际工作流」的小研究。如果你每天用 Claude Code 或 Cursor 写代码，我想请你做一个 3 天的轻量笔记（每次你切出 IDE 去问别人事情时记一条，每条就一句话）+ 一个 30 分钟的回放访谈。不是产品访谈、没有要卖你的东西，纯粹想搞清楚真实情况。有兴趣吗？

**B-E1 招募原话：**
> 我在做一项关于「生产环境里 agent 怎么和它不掌控的东西打交道」的开发者访谈。如果你有 LangChain/CrewAI/MCP/AutoGen 的 agent 在 prod 或接近 prod 跑，我想花 30 分钟听你讲最近一次你的 agent 调用了某个你不掌控的东西时发生了什么。纯发现、无 pitch。

## 3. C-E1 日记协议（3 天轻量日记）

### 3.1 设计原则

- **轻**：每条 = 一条 Slack 消息长度（≤ 2 句），目标是降低记录成本到「不打断心流」。
- **触发明确**：触发事件是「**你切出 IDE（alt-tab）去向一个人要东西**」。不是所有中断，只针对「向人要」。
- **零诱导**：日记模板**不提** agent、不提路由、不暗示「这本该更快」。只采集事实。
- **零工具依赖**：日记用受访者现成的低摩擦渠道（给自己发 Slack / 便签 / 一个共享表单）。**本研究不构建任何插件、不安装任何 MCP 工具、不需要任何 Confer 代码**——那是后续 C-E2 的 Wizard-of-Oz 范畴，本研究刻意停在「访谈 + 笔记」这一最便宜层。

### 3.2 给受访者的日记指引原话模板

> **接下来 3 个工作日，每次你从 IDE 切出去、为了向某个人要点什么（不是查文档/搜网/问模型，是问人），就记一条。每条就一句话，像发条 Slack 一样快。** 记这几样就够：
> 1. **大概几点**（不用精确）。
> 2. **你当时卡在什么上 / 想要什么**（一句话）。
> 3. **你去问的是谁**（用代号即可，如「后端 lead」「上一个写这块的人」「DevOps」，不用真名）。
> 4. **你最后怎么解决的**：Slack DM / 开会 / 当面问 / 放弃了 / 别的。
>
> 如果某次你本来想问人、但发现模型/搜索就够了，也记一条，标一句「其实模型能答」——这同样有用。**不要为了凑数而记，也不要漏记**。一天结束如果一条都没有，就回我「今天 0 条」，这也是有效数据。

### 3.3 日记字段（结构化，供后续打标）

每条日记落入如下结构（受访者随手记，研究者访谈时补全）：

| 字段 | 来源 | 备注 |
|---|---|---|
| `timestamp_rough` | 受访者 | 时段即可 |
| `blocker_text` | 受访者 | 卡点原话 |
| `target_person_role` | 受访者 | 代号，不要真名（PII） |
| `resolution_action` | 受访者 | Slack/会议/当面/放弃/其他 |
| `self_flag_model_solvable` | 受访者自标 | 「其实模型能答」标记 |
| `classification` | **研究者访谈时定** | code/lib/repo vs 真·跨人 tacit（见 §7） |
| `recurring` | 研究者访谈时定 | 是否反复出现（过去 2 周内多次） |

### 3.4 日记执行节奏

- Day 0：发指引 + 让受访者复述一遍触发条件（确认理解「只记向人要」）。
- Day 1–3：每天傍晚一条轻提醒（「今天有几条？0 条也回我」），**提醒文案不得**含产品暗示。
- Day 3 结束后 **24–48 小时内**排访谈（趁记忆新鲜），避免事后合理化。

## 4. C-E1 访谈脚本（30 分钟）

**结构：暖场 → 回放日记 → 逐条分类 → 反事实 → devil's-advocate。** 研究者（**role: interviewer**）只读**加粗/引号**的问句，括号内是追问/分支指引。

### 4.0 暖场（3 分钟）

> 「谢谢你做这几天的笔记。先说清楚：这不是产品访谈，我没有东西要卖你，今天我越听到『其实这事没什么大不了』越有用。你可以随时打断我、纠正我。我们就过一遍你记的那些条目。」

（建立 falsification-friendly 氛围：明确告诉受访者「否定也有价值」。）

### 4.1 回放日记（8 分钟）

逐条过。每条问：

> 1. 「**这条当时具体发生了什么？带我回到那个时刻——你在 IDE 里干到哪一步卡住了？**」
> 2. 「**你当时到底想从这个人脑子里拿到什么？**」（探针：是事实、是判断、是许可/凭据，还是「只有 ta 知道」的历史原因）
> 3. 「**这种情况，过去两周里还发生过几次？是一次性的还是反复的？**」（用于 `recurring` 标）

### 4.2 逐条分类（6 分钟）

对每条，研究者在心里走 §7 分类树，但**用中性问句**让受访者自己暴露类别，不要替 ta 归类：

> 4. 「**如果当时 Claude/Cursor 模型再强一点、或者你多搜一下文档/代码库/网，它能回答这个吗？还是说这东西只在那个人脑子里、文档里根本没有？**」
>    - 若「模型/文档能答」→ 倾向 **code/lib/repo**（model-solvable），打标。
>    - 若「只在那个人脑子里 / 是个决策为什么这么定的 / 是谁有权限 / 谁拥有这块」→ 倾向 **真·跨人 tacit need**，进入 4.3。
> 5.（仅对疑似 tacit 的条目）「**这属于下面哪种：(a) 某个决策当初为什么这么定的理由，(b) 谁有某个东西的 access/凭据/所有权，(c) 需要一个领域专家的口头判断，(d) 别的？**」（落入 §7 子类）

### 4.3 反事实：agent-routing vs Slack DM（6 分钟）

只对**真·跨人 tacit** 条目问。这是 C-v1 反事实部分的核心，决定 successCriteria 第二半。

> **设定提醒（对受访者口头给出，避免他脑补不存在的产品）：** 「下面让你设想的『那个人的 agent』，现在并不存在，纯属假设——我想知道的是，假如它存在，对你这件事有没有用。」

> 6. 「**你当时是怎么解决的？**」（对回日记的 `resolution_action`）
> 7. 「**当时那个人在线吗？你 Slack DM 一下大概多久能得到答案？**」（建立现成替代基线）
> 8. 「**设想一下：如果你不用切出去找人，而是在 IDE 里直接问『那个人的 agent』，它能基于那个人授权过的知识立刻给你一个够用的答案——这种情况下，这条会比你刚才那样 Slack/开会更快或更省事吗？**」
>    - 追问：「**快在哪？还是其实没快，因为你本来就要那个人亲自确认？**」（强制暴露「必须真人确认」的不可路由情形）
> 9. 「**有没有哪些条目，就算有这么个 agent，你还是会去戳真人？为什么？**」（采集反例，用于 §7 反事实打标的「否」）

### 4.4 devil's-advocate 探针（4 分钟）

主动替对立面说话，压低 vanity confirmation：

> 10. 「**说实话，你记的这几条里，是不是大部分其实搜一下、问下模型、或者在群里@一句就解决了？真正『非得是某个特定的人不可、而且文档里没有』的，到底有几条？**」
> 11. 「**很多人会说『反正那个人就在 Slack 上，我直接戳 ta 最快』——对你来说，让一个 agent 介入这件事，是帮忙还是反而多一层？**」
> 12. 「**如果我下周把这个『问某人 agent』的按钮给你，你诚实讲，你会真去用，还是大概率忘了它存在、继续 Slack？**」（口头 revealed-preference 预演——**仅采 stated 意向，真行为留给 C-E2**）

## 5. B-E1 访谈脚本（30 分钟，纯 problem interview，无 pitch）

**结构：是否真有跨 owner counterparty → identity 是否首要 blocker → 列全所有 blocker 并归类。** 固定脚本，逐字提问。

### 5.0 暖场（2 分钟）

> 「这不是 demo、不是 pitch，我没有产品要给你看。我想听你讲真实的生产情况，包括『其实没这个问题』这种答案——那对我反而最有用。」

### 5.1 是否真有跨 owner counterparty（8 分钟）

> 1. 「**带我看最近一次你的 agent 调用了某个你不掌控的东西。当时它在调什么？**」
> 2. 「**另一头是谁的 agent / 谁的服务？是你自己的另一个服务、一个你有 API key 的已知 SaaS、还是某个别人自主运行的 agent？**」
> 3. 「**粗略估一下，你的 agent 流量里，调『自家工具 / 你持有 key 的已知 SaaS』（intra-trust）大概占多少，调『别人自主运行的 agent』（cross-owner）大概占多少？**」（采集 `traffic_split`，§7 用）
> 4. 「**那次交互，有东西出问题/不放心吗？是什么？**」（开放，不引导向身份）

### 5.2 identity 是否首要 blocker（8 分钟）

> 5. 「**当你决定要不要信任另一头那个 agent 时，你具体在担心什么？**」（**完全开放，先不给选项**，让受访者自发说，记录顺序）
> 6.（受访者自发说完后再给框架）「**下面这几样，你最担心哪个、按严重程度排个序：**
>    - **(身份)** 我根本不确定对面到底是不是它声称的那个 agent；
>    - **(能力)** 它声称能干某事，但实际干不了 / 干得烂；
>    - **(责任)** 出了事谁负责、怎么追责；
>    - **(输出质量)** 它返回的内容对不对、可不可信；
>    - **(授权范围)** 它被授权能动哪些东西、动多了怎么办。」
> 7. 「**刚才排第一的那个，是不是你今天真正卡住、或者真踩过坑的？给我一个具体例子。**」（区分「理论担心」vs「真被咬过」）

### 5.3 列全所有 blocker 并归类（8 分钟）

> 8. 「**把你能想到的、阻碍你让 agent 跟外部 agent 打交道的所有障碍都列出来——技术的、信任的、组织的、商业的都算。**」（穷举）
> 9. 对每个 blocker，研究者归入 §7 的 `blocker_category`：**identity / capability / liability / output-quality / authorization-scope / other**。
> 10. 「**如果有人明天给你一个能可验证地证明『对面 agent 确实是它声称的那个』的东西——你今天会不会就去接？还是说就算身份解决了，前面那些（能力/责任/质量/授权）还是让你接不了？**」（直测 identity 的 necessary-but-not-sufficient 性）
> 11.（WTP 轻探针，不展开为 B-E4）「**你现在为这个问题付过钱、或正打算付钱吗？还是你预期它该是免费/开源标准自带的？**」（采集 `would_pay_now` Y/N，**仅记录，不进 B-E1 通过门**，归 B-E4）

### 5.4 收尾（4 分钟）

> 12. 「**有没有什么我该问却没问的？关于你的 agent 跟外部世界打交道这件事。**」
> 13. 「**最后诚实一句：『可验证身份』在你真实的待办清单上，是 top-3 的痛，还是其实排在很后面？**」

## 6. 反偏差守则（贯穿 C-E1 与 B-E1）

研究者在每场访谈前重读本节。违反任一条的条目在打标时降权或剔除。

1. **不诱导（no leading）**：先开放问，后给选项。§5.2 的 blocker 清单**必须**在受访者自发回答之后才出示，且每场必须等长读完所有选项（不得只强调「身份」）。任何「是不是……？」的封闭问句都要配一个「还是其实没有？」的对冲尾巴。
2. **不卖（no pitch）**：全程不出现产品名、idea 名、不演示、不描述「我们要做的东西」。受访者主动问「你在做什么产品」时，统一回答：「现在只是想搞清楚问题，产品的事以后再说。」
3. **不奖励确认（anti vanity-confirmation）**：访谈者对「是的，我超需要！」要追一句具体例子；对「其实没什么」要表达感谢而非失望。明确告诉受访者「否定的答案最有用」（§4.0/§5.0 已写入暖场）。
4. **强制证伪倾向（falsification-first）**：每个核心条目都配一个 devil's-advocate 反问（C-E1 §4.4，B-E1 §5.2 第7问 / §5.4 第13问）。**没有走完反问的条目不得标为「真阳」**。
5. **要 revealed，不要 stated**：优先采信「实际做了什么」（日记里的 `resolution_action`、prod 里真实流量）而非「会不会用/愿不愿付」的口头承诺。口头承诺单独记，权重低。**注意：本研究最强证据也只是「回忆 + 自报流量」，不是真行为；任何「会用某 agent」「愿付」都是 stated，必须明确标注并留给 C-E2/B-E4 用真行为复核。**
6. **隔离创始人回声**：访谈中若发现受访者其实是在转述创始人观点 / 之前和创始人聊过这个想法，立即把该条 `founder_proximity` 升级为 `direct` 并隔离。
7. **PII 守则**：日记与转录里人名一律用角色代号；不记录可识别个人的凭据/内部链接（与项目 PII 红线一致）。
8. **不脑补不存在的能力**：访谈中若引入「那个人的 agent」这类假设物，必须先声明它现在不存在（§4.3 设定提醒）。**不得**向受访者描述任何 telephony/email/浏览器自动化/陌生人发现/多租户登录等当前代码库没有的能力——这些是 greenfield，问到了如实说「还没有」。

## 7. 评分量规（rubric）

所有阈值**直接取自实验 successCriteria**，本节不引入任何新数字。

### 7.1 C-E1 编码规则

**Step 1 — 逐条日记分类（per-entry）：**

| `classification` | 判定规则 |
|---|---|
| `model_solvable`（code/lib/repo） | 受访者在 §4.2 第4问确认「模型/文档/搜索能答」**或**自标 `self_flag_model_solvable=true` |
| `cross_person_tacit` | 知识只在某特定人脑中/文档无、且落入子类 (a) 决策理由 / (b) access·凭据·所有权 / (c) 领域专家口头判断 / (d) 「谁拥有 X」 |
| `ambiguous` | 走完 §4.2 仍无法定——不计入任一侧分子，单列 |

**Step 2 — 参与者层面计 `genuine_recurring` 标志：**

> 一个参与者标 `genuine_recurring = true` 当且仅当：其日记中**至少 1 条** `classification=cross_person_tacit` **且** 该条 `recurring=true`（过去 2 周内多次）。

**Step 3 — 反事实打标（仅对 `cross_person_tacit` 条目）：**

| `routing_faster` | 判定规则（来自 §4.3 第8/9问） |
|---|---|
| `yes` | 受访者判断 agent-routing 比当时 Slack DM/会议更快或更省事，**且**说得出快在哪 |
| `no` | 必须真人亲自确认 / 没快 / 受访者明确说还是会戳真人 |

### 7.2 C-E1 通过判定（严格对齐 successCriteria：`>=40% … AND … >=1/3`）

设非创始人合格分母为 N（目标 12–15，`direct` 已隔离不计）。

- **条件 A（需求存在）：** `genuine_recurring=true` 的参与者 ÷ N **≥ 40%**（successCriteria 举例 6/15）。
- **条件 B（路由更快）：** 在满足条件 A 的那些参与者中，其 `cross_person_tacit` 条目里 `routing_faster=yes` 的占比 **≥ 1/3**。
- **C-E1 PASS = 条件 A AND 条件 B 同时成立。**

| 结果 | 触发 decisionRule（原文语义） |
|---|---|
| PASS | 跨人 JTBD 存在 → 推进 Wizard-of-Oz **C-E2** 测路由行为（注意：PASS 只是「需求信号存在」，**不**等于 bet 通过——bet 仍待 C-E3） |
| FAIL | **C-v1 被证伪**（IDE 需求是 model/web/Slack 可解）→ 降权 idea C，且整个 agent-to-agent portfolio 得 desirability 红旗 → **build 前先升级上报** |

> **敏感性检查（NON-FOUNDER GATE）：** 额外报「仅 `cold` 子样本」与「去掉 `network_1hop` 后」条件 A 是否仍 ≥40%。若整体过、但去掉 1hop 就跌破，标记为「依赖创始人圈」，视同未稳过，须扩冷样本复测。

### 7.3 B-E1 编码规则

**Step 1 — 每位受访者三个布尔标（来自 §5）：**

| 标 | 判定规则 |
|---|---|
| `counterparty_exists` | §5.1：受访者有**今天真实**的 cross-owner（别人自主运行的 agent）交互实例，非纯 intra-trust、非未来设想 |
| `identity_primary_blocker` | §5.2 第6问排序中 **identity 排第 1**，**且** §5.2 第7问能给出真踩过的具体例子（非纯理论担心） |
| `would_pay_now` | §5.3 第11问自报现在付/打算付（仅记录，**不进入 B-E1 通过门**，归 B-E4） |

**Step 2 — blocker 归类（per blocker，§5.3）：**

每个被列出的 blocker 归入 `identity / capability / liability / output-quality / authorization-scope / other`。汇总到全样本层面，算「**非身份类 blocker 占所有被点名 blocker 的比例**」。

### 7.4 B-E1 通过判定（严格对齐 successCriteria：`>=8 of 20 (40%) … AND <50% 非身份`）

设非创始人合格分母为 M（招募目标 20，`direct` 已隔离）。

- **条件 A（counterparty 真实 + 身份是首要）：** 同时满足 `counterparty_exists=true` **且** `identity_primary_blocker=true` 的受访者 **≥ 40%**，并对齐 successCriteria 的绝对量级 **8/20**（若实际合格 M≠20，按 40% 折算并同时标注绝对人数）。
- **条件 B（身份确为主因）：** 全样本中**非身份类 blocker 占所有被点名 blocker < 50%**。
- **B-E1 PASS = 条件 A AND 条件 B。**

| 结果 | 触发 decisionRule（原文，分三种 fail） |
|---|---|
| PASS | counterparty 问题真实且 identity-shaped → B 留在 Wave 1 → 推进 funnel 测试（B-E2） |
| FAIL（流量压倒性 intra-trust） | SDK 是「等市场的方案」→ 暂停 B，先把重心 reweight 到能制造 cross-owner 网络的 A/D，再卖身份 |
| FAIL（blocker 真实但**非** identity） | 把 wedge 转向被点名的那个真 blocker，再 build；同时按 shared identity assumption 重估 B/D/F/H 四个 idea 的身份前提 |

### 7.5 双盲编码与一致性

- 每条日记 `classification` 与每个 `blocker_category` 由**两名编码者（role: coder-A / coder-B）独立打标**；不一致条目进入仲裁（role: arbiter），仲裁未果归 `ambiguous`/`other`，**不计入任一侧分子**（防止把模糊样本算成真阳）。
- `identity_primary_blocker` 必须有「真踩过的具体例子」字段非空才成立——只填了排序、举不出例子的，降为「理论担心」，不算真阳。
- 单人团队（只有创始人一人）时，第二编码者可由**一名不参与本研究招募的外部同行**临时担任，确保 `classification` 不被创始人单方主观裁定。

## 8. 伦理与同意、记录方式、样本量与排期

### 8.1 知情同意与伦理

- 访谈与日记开始前取得**明确口头/书面同意**：说明用途（产品早期研究）、录音与否、可随时退出、数据如何保存与删除。
- **匿名化**：转录与日记中人名一律角色代号；不采集第三方个人的凭据、内部 URL、客户数据。与项目 PII 红线一致——不记录可识别的敏感个人信息。
- **无诱导性补偿**：可给小额谢礼（如礼品卡），但补偿**与「是否说出正面答案」无关**，且招募文案不得暗示「需要你确认某需求才付」。
- **创始人利益冲突披露**：对 `network_1hop` 受访者说明研究者即产品作者，降低「为照顾朋友而附和」的压力（明示「否定更有用」）。

### 8.2 记录方式

| 项 | 方式 |
|---|---|
| 日记 | 受访者自选低摩擦渠道（Slack DM 给自己 / 便签 / 共享表单）；研究者访谈时誊入结构化字段（§3.3）。**不建任何插件/工具。** |
| 访谈 | 征得同意后录音 + 转写；转写即时做角色代号匿名化 |
| 打标 | 每条目落入 §7 字段；双编码者独立打标 + 仲裁记录 |
| 审计轨迹 | `founder_proximity` 标 + 隔离清单单独存档，供下游 C-E2/C-E3 复用，证明 NON-FOUNDER GATE 已执行 |

### 8.3 样本量（全部来自 successCriteria，无新数字）

| 实验 | 招募目标 | 阈值分母依据 |
|---|---|---|
| C-E1 | **12–15** 名 Claude/Cursor 日活开发者；`cold` ≥ 1/3 | successCriteria 用 6/15 ≈ 40% |
| B-E1 | **20** 名 prod/near-prod agent builder；`cold` ≥ 1/3 | successCriteria 用 8/20 = 40% |

`direct`（创始人本人/直接同事）记录不计入上述分母，仅作试运行与脚本校准。

### 8.4 排期（对齐 timelineDays：C-E1 = 9 天，B-E1 = 12 天，可并行）

**C-E1（9 天）：**

| 天 | 活动 | 负责角色 |
|---|---|---|
| D1–2 | 招募 + screener 过滤；锁定 12–15 名合格者（含 cold 配额）；发日记 Day 0 指引 | recruiter / interviewer |
| D3–5 | 3 天日记执行（每日轻提醒，0 条也回收） | interviewer |
| D5–8 | 日记结束后 24–48h 内排完 30 分钟访谈；逐场即时转写匿名化 | interviewer |
| D8–9 | 双编码打标 + 仲裁 + 算条件 A/B + 敏感性检查；出 PASS/FAIL 与 decisionRule 触发 | coder-A / coder-B / arbiter |

**B-E1（12 天，可与 C-E1 并行）：**

| 天 | 活动 | 负责角色 |
|---|---|---|
| D1–3 | 多渠道招募（Discord / GitHub issue 线程 / r/LocalLLaMA / Confer 联系人）；screener 过滤至 20 名合格者（cold ≥ 7） | recruiter |
| D4–10 | 排 + 跑 20 场 30 分钟 problem interview；并行采集每位的 `traffic_split` 自报 | interviewer |
| D10–12 | 双编码 blocker 归类 + 算条件 A/B（≥8/20 量级且非身份 <50%）；出 PASS/FAIL 与三分支 decisionRule；按 shared identity assumption 标注对 D/F/H 的连带含义 | coder-A / coder-B / arbiter |

### 8.5 交付物（研究结束时）

1. C-E1 / B-E1 各一份打标数据表（per-entry + per-participant）。
2. `founder_proximity` 隔离审计清单——证明 NON-FOUNDER GATE 已执行（`direct` 记录已排除出分母），供 C-E2/C-E3 接续。
3. 一页结论：每个实验的条件 A/B 数值、PASS/FAIL、触发的 decisionRule 原文、敏感性检查结果，以及（B-E1）对 B/D/F/H 身份前提的连带判读。
4. **范畴声明（必含）：** 「本研究只产出 stated/recalled desirability 信号，**未、且不能单独验证全局 bet**；全局 bet 的通过判定属 C-E3（≥2 个独立非创始人完成 unscripted consequential agent-to-agent 往返）。C-E1/B-E1 的任何 PASS 均不得被解读为 bet 通过。」
